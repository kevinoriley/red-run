# Sliver C2 MCP Server

MCP server providing gRPC integration with the [Sliver C2 framework](https://github.com/BishopFox/sliver).
Connects to a running Sliver server using mTLS operator credentials and exposes
implant management, command execution, file transfer, pivoting, and
post-exploitation as MCP tools.

## Prerequisites

### Sliver server

A running Sliver server with at least one operator config generated:

```bash
# In the Sliver console:
new-operator --name red-run --lhost <server-ip>
```

This produces a `.cfg` file containing mTLS certificates.

### Protobuf compilation

Proto stubs must be compiled before first use:

```bash
scripts/update-sliver-protos.sh
```

This downloads `.proto` files from the Sliver repo and compiles them into
`tools/sliver-server/proto_gen/`.

### Python dependencies

```bash
uv sync --directory tools/sliver-server
```

## Configuration

The server locates the Sliver operator config in this order:

1. **`SLIVER_CONFIG`** environment variable — path to a `.cfg` file
2. **`engagement/config.yaml`** — `c2.sliver_config` key
3. **`~/.sliver-client/configs/default.cfg`** — Sliver client default

Example `engagement/config.yaml`:

```yaml
c2:
  sliver_config: /path/to/operator.cfg
```

The `.cfg` file is JSON containing the operator name, mTLS certificates,
and server address/port. Generated by `new-operator` in the Sliver console.

## Usage

The server runs as an MCP server, started by Claude Code via `.mcp.json`:

```bash
uv run --directory tools/sliver-server python server.py
```

Connection is lazy — the gRPC channel is established on the first tool call,
not at server startup.

## Tools

### Tier 1 — Core Operations

| Tool | Parameters | Description |
|------|-----------|-------------|
| `sliver_sessions` | (none) | List all active sessions with metadata |
| `sliver_execute` | `session_id`, `exe`, `args`, `output` | Execute command on target |
| `sliver_upload` | `session_id`, `local_path`, `remote_path` | Upload file to target |
| `sliver_download` | `session_id`, `remote_path`, `local_path` | Download file from target |
| `sliver_ls` | `session_id`, `path` | List directory on target |
| `sliver_ps` | `session_id` | List processes on target |
| `sliver_ifconfig` | `session_id` | List network interfaces on target |
| `sliver_netstat` | `session_id` | List network connections on target |

### Tier 2 — Implant Generation & Listeners

| Tool | Parameters | Description |
|------|-----------|-------------|
| `sliver_generate` | `os`, `arch`, `format`, `c2_endpoints`, `name`, `debug`, `evasion` | Generate implant binary |
| `sliver_listeners` | (none) | List active listeners (jobs) |
| `sliver_start_listener` | `type`, `host`, `port` | Start mTLS/HTTPS/HTTP listener |
| `sliver_stop_listener` | `job_id` | Stop a listener |

### Tier 3 — Pivoting & Routing

| Tool | Parameters | Description |
|------|-----------|-------------|
| `sliver_pivot_start` | `session_id`, `type`, `bind_address` | Start pivot listener on target |
| `sliver_pivot_stop` | `session_id`, `listener_id` | Stop pivot listener |
| `sliver_pivot_graph` | (none) | Show implant routing topology |
| `sliver_socks_start` | `session_id`, `port` | Start SOCKS5 proxy through session |
| `sliver_socks_stop` | `session_id` | Stop SOCKS5 proxy |
| `sliver_portfwd_start` | `session_id`, `remote_host`, `remote_port`, `local_port` | Start port forward through session |
| `sliver_portfwd_stop` | `session_id`, `forward_id` | Stop port forward |

### Tier 4 — Post-Exploitation

| Tool | Parameters | Description |
|------|-----------|-------------|
| `sliver_execute_assembly` | `session_id`, `assembly_path`, `args`, `process`, `am_si_bypass`, `etw_bypass` | In-memory .NET assembly execution |
| `sliver_sideload` | `session_id`, `dll_path`, `entry_point`, `process`, `args` | Load native DLL/SO into target process |
| `sliver_make_token` | `session_id`, `username`, `password`, `domain` | Create Windows logon token |
| `sliver_get_system` | `session_id` | Elevate to SYSTEM |
| `sliver_screenshot` | `session_id` | Capture target screen |

### Tier 5 — Beacons

| Tool | Parameters | Description |
|------|-----------|-------------|
| `sliver_beacons` | (none) | List all active beacons |
| `sliver_beacon_tasks` | `beacon_id` | List beacon task queue |
| `sliver_beacon_interactive` | `beacon_id` | Upgrade beacon to interactive session |

## Architecture

```
server.py          → FastMCP tool definitions, response formatting
connection.py      → gRPC channel management, mTLS auth, config resolution
proto/             → Vendored .proto files from Sliver repo
proto_gen/         → Compiled Python stubs (gitignored, built by script)
```

The server uses lazy initialization throughout:
- gRPC channel created on first tool call, cached for subsequent calls
- Proto modules imported at first use with clear error if not compiled
- atexit handler closes the channel on shutdown

All tools return structured dicts serialized by FastMCP. Errors are returned
as `{"error": "message"}` rather than raising exceptions, so agents get
actionable error messages.

Evidence (screenshots, downloaded files, assembly output) is saved to
`engagement/evidence/` when the engagement directory exists.
